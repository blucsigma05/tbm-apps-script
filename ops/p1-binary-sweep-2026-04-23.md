## Observations not tied to a specific backlog row

- **Empty `<title>` bug (items 43 + 44):** `/wolfdome` and `/sparkle-kingdom` both return `document.title === ""` despite rendering their body content correctly. Shared symptom suggests a common template or boilerplate. Not escalated to a FAIL Issue this sweep because neither violates its backlog done-when; worth a follow-up sweep task.
- **Loading-state fragility (items 29, 30, 32, 44, 48, 49):** many education/dashboard surfaces show loading placeholders for 5–12s before content renders. The 3.5s first-pass wait was too short for half of them. Only `/sparkle` (item 32) failed to resolve after 15s — that is the clean signal that the other surfaces are slow-but-correct. A P2 item to cap loader timeouts + show explicit error states (rather than indefinite spinners) would harden the JJ-age UX.
- **`/homework` is a Daily Missions alias on reading+creative days:** the route did not render a math/science question surface on Thursday; it rendered the Daily Missions launch page (since Thursday's schedule has no math/sci). The backlog done-when for item 31 assumes a direct homework-question render, which does not match today's architecture. Either the backlog needs re-scoping (per-day-of-week verification) or the routing needs to branch between homework-today and missions-today.
- **Label schema gap:** filed FAIL Issues use `kind:bug` + `severity:major` + `model:sonnet`. CLAUDE.md documents an `area:<jj|buggsy|education|finance|shared>` family, but only `area:infra` currently exists on Gitea — area labelling was not applied to avoid adding labels out of scope for this sweep.

## Method notes

- **No production mutations.** For buttons that would write to `KH_` sheets (`CLAIM RINGS`, `I Did It!`, `LAUNCH` → submit), the sweep verified button presence + click-ability in the DOM only; no clicks were fired on the live system. Write-path verification belongs in a separate audit.
- **PIN gate (45, 46) not exercised.** LT's PIN was not provided to this session; the sweep verified the gate redirects correctly but did not cross it.
- **Fire Stick surfaces (47, 48) not exercised on-device.** Chrome-at-viewport is a proof-of-life; it is not equivalent to a Fully Kiosk render on the actual Fire Stick hardware (Android WebView, ES5 constraints, older stack). Per prompt, these are INCONCLUSIVE until LT/JT verify on the devices.
